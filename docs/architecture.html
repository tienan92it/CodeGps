Architecture

Six layers, two databases, one runtime contract. Syntax is deterministic. Meaning is agent-driven.

Layered model

The layers map loosely onto the DIKW pyramid: raw data at the bottom, derived meaning at the top. Each layer has its own SQLite table family. Edges cross layers explicitly via foreign keys.

LayerContentProduced by

L1 Conversationssessions, turns, tool callsdeterministic (file parsers)

L3 Conceptsclustered facts with names + structured summariesagents (Clusterer · Summarizer)
L4 Cross-Projectshared concepts between reposmechanical (exact + SimHash) + agent (Linker)

Pipeline

Each ingest run pulls new turns, segments them into windows, and routes each window through the deterministic syntax pass and the agent stack. Inputs flow top-down; outputs accumulate as edges back into earlier layers.

flowchart TB raw[L1 raw turns] --> seg[Window Segmenter] seg --> syn[Syntax Pass] seg --> tri[Triage Agent] tri -->|kept| ded[Dedupe Agent] ded -->|novel| route{Route by domain} route --> dec[Decision] route --> biz[BusinessLogic] route --> int[Intent] route --> ps[ProblemSolution] syn --> facts[L2 facts] dec --> facts biz --> facts int --> facts ps --> facts facts --> res[Code Resolver] facts --> clu[Clusterer] clu --> sum[Summarizer] sum --> concepts[L3 concepts] concepts --> link[Linker + mechanical] link --> global[L4 global] facts --> ver[Verifier]

Deterministic stages

  • Window Segmenter — slices a session into windows on user-turn boundaries with size and time caps.
  • Syntax Pass — extracts artifacts that are unambiguous from form: paths, code blocks, shell commands, error messages, stack traces, URLs, ticket IDs.
  • Code Resolver — for any mention (syntax or agent), look it up in code.db and write a k_to_code edge.

Agent stages

  • Triage — drops noise and labels each window on four axes.
  • Dedupe — embedding similarity against prior windows and prior facts.
  • Extractors — Decision, BusinessLogic, Intent, ProblemSolution. Routed by triage domain so business-logic agents don't waste tokens on chitchat.
  • Clusterer — for each new fact, picks attach / create / merge against top-K nearest concepts.
  • Summarizer — names and writes a structured summary for each touched concept.
  • Linker — proposes cross-project links the mechanical pass missed.
  • Verifier — periodic sweep for contradictions, supersessions, low-confidence pruning.

The agent layer

There are nine chat agents and one embedding-only agent. Each has a versioned prompt, a JSON schema, and a model reference resolved from config. All share a single runtime that handles caching, validation, and persistence.

Agent runtime

Every agent call goes through one function. Inputs are a typed payload plus the agent's prompt and schema; outputs are validated JSON, cached by content hash, and persisted to agent_runs for audit. 

interface Agent<I, O> {
  name: string;
  promptVersion: number;        // bump invalidates cache
  modelKey?: string;            // overrides config lookup
  schema: JsonSchema;           // strict validation
  prompt(input): ChatMessage[];
  postprocess?(o, input): { output, confidence };
}

interface AgentRuntime {
  run<I, O>(agent, input): Promise<{ output, confidence, model, cached }>;
  // 1. hash(agent.name + model + prompt_version + input) -> cache lookup
  // 2. call backend (Ollama / OpenAI-compatible / Anthropic)
  // 3. parse JSON; validate against schema; one repair retry on failure
  // 4. write to agent_runs (success or failure)
}

Cache invariants

  • Same input + same model + same prompt version → cached hit, zero cost
  • Bump promptVersion → all prior runs ignored, agent re-runs
  • Swap model in config → cache miss only for affected agent; others unchanged

Three databases, three responsibilities. The split keeps the code DB drop-in for codegraph users and lets the knowledge schema evolve independently.

Design trade-offs

  • Local-only. No cloud sync. The hard runtime dependency is a local (or pluggable) LLM. The alternative would be regex-based NL understanding, which drifts the moment anyone tries to maintain it.
  • Cached, not deterministic. Agent output is stable per (agent, model, input_hash, prompt_version). Bumping the prompt version invalidates the cache by design.
  • Incremental clustering. The Clusterer attaches greedily and may drift; codegps link --rebuild triggers a full recompute when the drift becomes visible.
  • Pull-based ingest. No real-time chat hooks; re-run codegps ingest (or call codegps_ingest via MCP) to pick up new sessions.
  • Code DB stays codegraph-compatible. New code-side fields go in project_metadata JSON rather than schema changes.
  • Conservative triage by default. Windows are kept unless an axis is firmly negative with confidence ≥ 0.6. triage audit --dropped lets you tune.
